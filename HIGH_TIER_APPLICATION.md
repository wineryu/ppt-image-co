# 高额度申请版材料

## 表单第 04 项

我在做一个叫 `PPT Image First` 的小工具，目标是把 AI 生成 PPT 的流程从“先写一堆文字”改成“先确定每一页要让观众看到什么”。现在很多 AI PPT 工具能很快生成大纲，但落到页面上经常变成文字堆叠：标题、三四个要点、配一张并不准确的图。我的思路是先把每一页拆成视觉锚点、页面目标、短文案和图片提示词，再继续交给 Agent 迭代。

目前我已经完成了一个可运行的静态原型。用户输入主题、听众、页数和视觉风格后，工具会生成一组分镜卡片，每张卡片包含页面目标、视觉方向、页面文案、图片生成提示词和标签，并支持导出 Markdown 与 JSON。这个结果可以继续用于生成图片、写 README、接入 PPTX 导出，或者作为后续 Agent 工作流的中间格式。

我使用 Codex 的方式不是只让它补全几行代码，而是把它放进完整开发流程里：先讨论产品目标和申请场景，再搭建前端原型、设计页面结构、实现导出逻辑、整理 README 和申请材料。后续我计划继续加入 MiMo API 调用，让它根据不同演示类型自动生成更贴合的分镜；再加入图片生成、PPTX 导出、截图检查和模板库。

这个项目需要较高 token 额度，主要因为它会反复处理长上下文：一边要读项目代码和导出格式，一边要参考页面截图、视觉风格要求、演示主题和用户反馈。真正做成可用工具后，每次生成一套 PPT 都可能经历多轮：主题理解、分镜生成、图片提示词优化、页面文案压缩、截图检查、导出结构修正。高额度能让我把它从一个静态原型推进到完整的 Agent 辅助 PPT 创作流程，而不是只停留在一次性 demo。

## 更短版本

我在做 `PPT Image First`，一个视觉优先的 AI PPT 分镜工具。它不直接从文字大纲生成 PPT，而是先把主题拆成每页的视觉锚点、页面目标、短文案和图片提示词，再导出 Markdown/JSON 供 Agent 继续迭代。目前已有可运行静态原型，后续计划接入 MiMo API、图片生成、PPTX 导出和截图检查。这个项目会大量消耗长上下文，因为每次迭代都要同时参考代码、页面截图、视觉要求、导出结构和用户反馈，所以希望获得较高 token 额度来持续完成完整工作流。
